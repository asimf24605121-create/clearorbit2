==================================================
AUDIT FINDINGS YOU MUST USE
==================================================

These are the major issues found and must be fixed:

1. 100-USER CAP
The Users table currently relies on ALL_USERS from admin_init with take: 100.
Admins become blind beyond 100 users.
Search, filters, export, and actions are all incomplete because they only operate on the first 100 users.

2. TWO EXPIRY SYSTEMS CONFLICT
User.expiryDate and UserSubscription.endDate exist separately and drift out of sync.
The table/badge/status uses User.expiryDate, while real access lives in UserSubscription.
This makes expiry/status inaccurate after subscription extensions.

3. ACTIVE PLATFORM COUNT IS WRONG
The code counts subscriptions with isActive=1 but does not check whether endDate has already passed.
Expired subscriptions may still show as active platforms.

4. CREATE USER FLOW CAN SILENTLY UPDATE AN EXISTING USER
If username exists, create_user_with_sub silently switches into update mode and only modifies subscriptions.
This is dangerous and confusing.

5. EDIT MODAL CANNOT REVOKE INDIVIDUAL PLATFORM ACCESS
Admins can add platforms but cannot remove/deactivate a specific user subscription cleanly.

6. DISABLE DOES NOT REVOKE SESSIONS
Disabling a user flips isActive but does not terminate active sessions or revoke access immediately.

7. DELETE IS TOO DANGEROUS
Delete uses weak confirm() and cascades everything.
No impact summary, no type-to-confirm, no pre-check for pending payments.

8. STATUS SEMANTICS ARE INCONSISTENT
The filter and badge thresholds differ.
A user may be classified as expiring in one place and active in another.
There is no "Partial Access" concept.

9. HEAVY FULL REFRESH
After every mutation, loadStats() refetches a heavy admin payload.
This is inefficient at scale.

10. CSV EXPORT IS WRONG / INCOMPLETE
Client-side export only covers the loaded array and silently caps at 100 users.
